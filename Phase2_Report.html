1. Executive Summary

Phase 2 delivers a fully functional backend for the National Emergency Response Coordination Platform, a distributed system designed to manage emergency incidents, dispatch responders, and provide real-time operational analytics for Ghana's emergency services.

Key deliverables in Phase 2:

  • Event-driven architecture using RabbitMQ topic exchange (emergency_platform)
  • Real-time vehicle location tracking via Socket.IO
  • Self-sufficient analytics service powered by consumed events (no upstream HTTP calls)
  • Station registry for police stations, fire stations, and hospitals
  • Swagger UI documentation on every service
  • Single shared .env configuration file
  • MongoDB Atlas with per-service database isolation

2. System Architecture

2.1 Microservices Overview

ServicePortDatabaseResponsibility
Auth Service 3001 auth_service User registration, login, JWT issuance & refresh, role management
Incident Service 3002 incident_service Emergency incident lifecycle, automatic responder matching using Haversine distance
Dispatch Service 3003 dispatch_service Vehicle fleet management, GPS tracking, station registry, Socket.IO real-time broadcasts
Analytics Service 3004 analytics_service Operational dashboards, KPI metrics, incident mapping to Ghana regions — all from local DB populated by RabbitMQ events

2.2 Communication Patterns

The system uses two communication patterns:

  • Synchronous HTTP (REST) — Used by the Incident Service to query the Dispatch Service for available vehicles when auto-dispatching. Fast, request/response.
  • Asynchronous RabbitMQ events — All cross-service state propagation (incident creation, dispatch confirmation, vehicle status, analytics). Decoupled, durable, and resilient to service downtime.

4.1 RabbitMQ Messaging Layer

File: common/messaging/rabbitmq.js

A singleton connection manager for RabbitMQ with automatic retry and reconnection logic. All four services import this module — it manages a single channel and re-uses it across publish/subscribe calls.

Key behaviours:

  • Retry on startup — up to 10 attempts with 3-second delays before giving up
  • Auto-reconnect — listens to connection.on('close') and schedules reconnect
  • Durable topic exchange — declared on every connection so it survives broker restarts
  • Prefetch(10) — limits concurrent unacknowledged messages per consumer
  • ACK/NACK — messages are acknowledged on success and nacked (no requeue) on handler error

4.3 Utility Helpers

File: common/utils/helpers.js

Haversine Distance Formula

Used by the Incident Service to rank vehicles by proximity before dispatching the nearest available one. Returns distance in kilometres between two GPS coordinates. 

const calculateDistance = (lat1, lon1, lat2, lon2) => {
  const R = 6371; // Earth's radius in km
  const dLat = toRad(lat2 - lat1);
  const dLon = toRad(lon2 - lon1);
  const a = Math.sin(dLat/2)**2 +
            Math.cos(toRad(lat1)) * Math.cos(toRad(lat2)) * Math.sin(dLon/2)**2;
  return R * 2 * Math.atan2(Math.sqrt(a), Math.sqrt(1 - a));
};

6. Incident Service (Port 3002)

6.1 Responsibilities

  • Full lifecycle management for emergency incidents (Created → Dispatched → In Progress → Resolved)
  • Automatic nearest-responder matching using Haversine distance against the Dispatch Service vehicle list
  • Publishing lifecycle events to RabbitMQ for downstream services to react
  • Subscribing to vehicle.status.changed to maintain awareness of vehicle availability

6.2 API Endpoints

MethodPathAuthDescription
POST /incidents JWT Create a new incident and trigger automatic responder dispatch.
GET /incidents/open JWT List all unresolved incidents sorted by creation date.
GET /incidents/all JWT Paginated list of all incidents. Query: page, limit.
GET /incidents/:id JWT Get single incident by ID.
PUT /incidents/:id/status JWT Update incident status. Auto-calculates responseTime (minutes) on resolution.
PUT /incidents/:id/assign JWT Manually trigger responder matching for an existing incident.

6.3 Auto-Dispatch Logic

When an incident is created, matchAndDispatchResponder() runs asynchronously:

  1. Maps incident type → vehicle type (e.g. fire → fire_truck)
  2. Queries Dispatch Service: GET /vehicles?type=fire_truck&status=available
  3. Computes Haversine distance from incident to each vehicle
  4. Selects the closest vehicle and updates the incident's assignedUnit
  5. Publishes INCIDENT_DISPATCHED — Dispatch Service marks vehicle busy, Analytics records event

7. Dispatch Service (Port 3003)

7.1 Responsibilities

  • Emergency vehicle fleet registry and real-time GPS tracking
  • Socket.IO server for live vehicle location push to connected frontend clients
  • Station registry (police stations, fire stations, hospitals with bed capacity)
  • Consumes incident.dispatched events and auto-marks vehicles as dispatched
  • Publishes vehicle location and status change events to RabbitMQ

7.2 Real-Time Architecture

The Dispatch Service wraps Express inside a Node.js http.createServer() and passes the same server instance to Socket.IO. This allows both REST and WebSocket traffic on port 3003. 

const server = http.createServer(app);
const io = new SocketIOServer(server, { cors: { origin: '*' } });
setIO(io); // injects io into vehicle routes

// On vehicle location update:
io.emit('vehicle:location', { vehicleId, latitude, longitude, status, timestamp });

// On incident.dispatched RabbitMQ event:
io.emit('vehicle:dispatched', { vehicleId, incidentId, vehicleType, distanceKm });

8. Analytics Service (Port 3004)

8.1 Design Philosophy — Event-Sourced Analytics

The Analytics Service is entirely decoupled from other services at query time. It consumes RabbitMQ events as they happen and maintains its own aggregated view of the world in analytics_service MongoDB. All dashboard and reporting queries hit local collections — no upstream HTTP calls, no auth dependency, no coupling to other services' databases.

Key architectural benefit If the Incident Service or Dispatch Service goes offline, the Analytics Service continues to serve cached metrics from its last known state. It will catch up automatically when the services come back online and resume publishing events.

8.2 Event Consumer

File: services/analytics-service/src/messaging/consumer.js

Subscribes to two queues at startup. For every event received:

  1. Persists the raw event to AnalyticsEvent collection (full audit log)
  2. Upserts a summarised document in IncidentSummary or VehicleSummary

The /analytics/incidents-by-region endpoint classifies each incident by its GPS coordinates into one of Ghana's 16 administrative regions using predefined latitude/longitude bounding boxes. Incidents outside all bounding boxes are classified as Other.

10. Bugs Fixed

Bug 1 — Route Shadowing (incident-service) GET /incidents/open was declared after GET /incidents/:id, causing Express to treat "open" as a document ID and return 404 or a CastError.
Fix: Moved all static routes (/open, /all) before the /:id parameterized route.
Bug 2 — Use-Before-Define in Status Update In PUT /:id/status, responseTime was calculated using incident.createdAt before the variable incident was assigned (it was set by the result of findByIdAndUpdate).
Fix: Fetch the incident with findById first, compute responseTime, then update.
Bug 3 — Responder Type Enum Mismatch getResponderType() returned 'police' and 'fire' but the Vehicle model enum uses 'police_car' and 'fire_truck'. Vehicle queries always returned empty arrays for police and fire incidents.
Fix: Updated the type map in helpers.js to use correct enum values.
Bug 4 — Analytics Service Unauthenticated HTTP Calls Original analytics routes made HTTP GET calls to Incident/Dispatch service endpoints which require JWT authentication, always receiving 401.
Fix: Analytics service now queries its own local MongoDB collections, populated via RabbitMQ event consumption. Zero upstream HTTP dependency.
Bug 5 — MongoDB URI Corruption Original code appended /dbname directly to a URI that already had /?appName=... query params, producing a malformed URI.
Fix: Removed the URI manipulation entirely; uses the dbName option in mongoose.connect() instead.
Bug 6 — Node.js v25 DNS SRV Resolution Failure Node.js v25 uses c-ares DNS resolver which cannot perform SRV lookups on Windows, causing querySrv ECONNREFUSED for mongodb+srv:// URIs.
Fix: Replaced mongodb+srv:// URI with a standard mongodb:// URI using all three Atlas shard hostnames and the replicaSet parameter discovered from DNS TXT records.
